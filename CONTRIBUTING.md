# Contributing to naba

Thanks for your interest in `naba` — a single-binary Rust CLI for AI image
generation across multiple providers (Google Gemini, OpenRouter, AWS Bedrock),
which also runs as a stdio MCP server.

`naba` is built almost entirely with AI agents (Claude Code). We **welcome
AI-assisted contributions** — the bar is not *how* you wrote a change, but that
you understand it, you stand behind it, and it passes the full validation suite.
See [AI-assisted contributions](#ai-assisted-contributions) below.

- **Issues & discussion:** <https://github.com/dixson3/naba/issues>
- **License:** MIT (see [`LICENSE`](LICENSE)). By contributing you agree your
  contribution is licensed under the same terms.

## How the software is constructed

A quick orientation so you know where things live and what is authoritative.

### Language & toolchain

- **Rust** (edition 2021), a single crate producing the `naba` binary
  (`src/main.rs`). Standard `cargo`; no unusual toolchain. CI builds on stable.
- **Near-zero-dep posture.** Dependencies are deliberately minimal: `clap`,
  `serde`/`serde_json`, `serde_norway` for YAML (**`serde_yml` is forbidden** —
  RUSTSEC-2025-0068), `reqwest` with **rustls-tls only**, `tokio`, `rmcp` (MCP),
  `include_dir` (compile-time skill embedding), `aws-sigv4` (Bedrock), and pure-Rust
  `flate2`+`tar`+`self-replace` for self-update. Prefer not adding dependencies;
  justify any new one in the PR.
- Tests use the standard Rust harness plus `wiremock` for HTTP mocking — no extra
  test frameworks.

### Repo layout

| Path | What it is |
|:--|:--|
| `src/` | The CLI. Submodules: `provider/` (gemini, openrouter, bedrock, `registry`, `select`), `self_cmd/` (self update/install), `mcp.rs` (MCP server), `skills.rs`/`embed.rs` (embedded skill tree), `config.rs`, `commands.rs`, `cli.rs`, `error.rs`, `output.rs`. |
| `tests/parity/` | Black-box parity/regression suite (Python + pytest, fully mocked — **no API keys needed**). Drives the built binary and snapshots stdout/stderr/exit code/files/HTTP. |
| `docs/specifications/` | **Authoritative UX contract**, split per-domain (`commands.md`, `providers.md`, `configuration.md`, `mcp.md`, `skills.md`, `distribution.md`, …; `README.md` is the index). Stable, append-only clause IDs (`SPEC-<AREA>-NNN`). `SPEC.md` at the repo root is now a redirect stub. |
| `DRIFT-CHECK.md` | Manifest declaring which docs must stay in agreement with the code (skill files, split specs, README, website). |
| `skills/naba/` | The embedded Claude Code skill (`/naba <subcommand>`), compiled into the binary. |
| `docs/plans/` | Sequenced archive of implementation plans. |
| `web/` | Pelican static site for naba.ysapp.net (S3 + CloudFront). |
| `.beads/` | Local issue database (see [Issues](#submitting-issues)). |
| `.github/workflows/` | `ci.yml`, `release.yml` (cargo-dist, **autogenerated**), `web-deploy.yml`. |

### Spec-driven development (important)

`naba` is **spec-first**. The per-domain specs under `docs/specifications/` are the
source of truth; the code implements them and the parity suite proves it.

- **Every behavior change updates the relevant `docs/specifications/*.md` file** —
  clause IDs are append-only, never renumbered — **and adds or adjusts a parity
  test** that cites the clause.
- `tests/parity/check_traceability.py` enforces that **every `[PINNED]` / `[NEW]`
  SPEC clause maps to a test** (a parity case's `spec:` field, a test module, or a
  justified entry in `traceability_exemptions.yaml`). A change that adds a clause
  without coverage fails the build.
- `DRIFT-CHECK.md` requires derived docs (the skill, the IG, the README, the
  website) to stay consistent with the code they describe — e.g. every CLI verb a
  skill file invokes must map to a real `clap` command, and the default-model id
  must agree across code and docs.

If your change conflicts with the spec, update the spec in the same PR (and say so
in the description). If it conflicts with an *implementation plan*, raise it in an
issue first.

### Releases & deploy

- Pushing a version tag (`vX.Y.Z`, matching the crate version in `Cargo.toml`)
  triggers cargo-dist: 4-platform binaries, shell + Homebrew installers, checksums,
  a GitHub Release, and a Homebrew formula push. `release.yml` is **autogenerated** —
  change `[workspace.metadata.dist]` in `Cargo.toml`, not the workflow.
- A successful release auto-deploys the website and re-mirrors `install.sh`.
- You do not need release access to contribute — the maintainer cuts releases.

## Getting set up

```bash
git clone https://github.com/dixson3/naba
cd naba
cargo build --release        # or: make build  (also copies ./naba)
```

The parity suite uses [`uv`](https://docs.astral.sh/uv/) to run its Python harness
(`tests/parity/`). No API keys or secrets are required — every provider call in the
suite is mocked.

## The validation gate

**Every contribution must pass the full local validation recipe before you open a
PR.** This is the real filter for quality, and CI runs the same checks:

```bash
cargo fmt --check
cargo clippy --all-targets -- -D warnings
cargo build --release
cargo test
uv run tests/parity/check_traceability.py
NABA_BIN="$PWD/target/release/naba" uv run --project tests/parity pytest tests/parity
```

(`make fmt lint test parity traceability` wrap these.) If a change intentionally
alters observable output, re-capture the golden snapshots
(`pytest … --update-golden`) and review the diff — don't weaken a test to make it
pass.

### Conventions

- **No secrets, ever.** Never commit API keys, AWS account ids, ARNs, IAM/profile
  names, Route53 zone ids, or GA ids. Account-specific values come from the
  environment (e.g. `NABA_HOSTED_ZONE_ID`, `NABA_GA_MEASUREMENT_ID`) and fail closed
  when unset. Generated id-bearing files are gitignored.
- **Exit codes are semantic** (`error::AppError { code, message }`): 0 ok, 2 usage,
  3 auth, 4 rate-limit, 5 api, 10 file-io. Preserve them.
- Match the surrounding code's idioms, error handling, and comment density.
- Keep unit tests network-free (seams + `wiremock`); isolate filesystem via temp
  dirs and env via a module-local mutex.

### Branch & commit

- Fork (or branch), keep changes focused, and open a PR against `main`.
- Write clear commit messages: a human-authored summary of **what and why**.
- One logical change per PR where practical; explain any new dependency.

## Submitting issues

Issues live at <https://github.com/dixson3/naba/issues>.

**A good bug report includes:** what you ran (exact command), what you expected,
what happened (stderr + exit code), your OS/arch, and how you installed `naba`.
Reproduction steps beat prose — a minimal repro is the fastest path to a fix.

**A good feature request** states the problem you're trying to solve, not just a
proposed implementation.

### Security issues

Please report suspected vulnerabilities privately to the maintainer rather than in
a public issue, with a **real, reproducible, independently verifiable** problem in
actual `naba` source. See the note on AI-generated security reports below.

## AI-assisted contributions

`naba` is itself built with AI agents, so we welcome AI-assisted work. The rules
below exist to keep quality high and maintainer time protected — they follow the
emerging cross-project consensus for 2025–2026 (see, e.g., the living
[open-source AI contribution policy catalog](https://github.com/melissawm/open-source-ai-contribution-policies),
[Kubernetes' policy](https://www.kubernetes.dev/docs/guide/pull-requests/#ai-guidance),
and [the Linux Foundation's generative-AI policy](https://www.linuxfoundation.org/legal/generative-ai)).

**The rule that matters most — human accountability.** By opening a PR or issue you
assert that **you understand the change, can explain and defend it in review, and
can maintain and debug it without the AI.** AI assistance does not transfer
responsibility; if your name is on the contribution, you own it. This — plus the
validation gate above — is what lets us welcome AI contributions safely.

**Do:**

- **Run the full [validation gate](#the-validation-gate) yourself** before
  submitting. Unvalidated AI output — code the author didn't run, read, or test —
  will be closed. The suite, not the disclosure line, is the real filter.
- **Disclose AI assistance** in your commits via a trailer. `naba`'s convention is
  `Co-Authored-By: Claude <noreply@anthropic.com>` for Claude-assisted work; a
  generic `Assisted-by: <tool/model>` trailer is also fine. Disclosure is
  *recommended*, not gatekept — it just helps reviewers calibrate.
- **Keep the "why" human.** Write your own PR description and commit summaries
  explaining intent; don't paste unreviewed AI narration.
- **AI agents:** read [`AGENTS.md`](AGENTS.md) — it carries the machine-readable
  build, spec, and workflow rules for this repo.

**Don't:**

- **Drive-by AI PRs** — large speculative refactors nobody asked for, or changes
  that don't pass the suite.
- **AI-generated issue or bug-report spam** — reports that "sound technical but
  contain nothing useful." Confirm an issue is real and reproducible before filing.
- **Unverified AI security reports.** Any security report prepared with AI must
  disclose that, and must describe a real, reproducible, independently verifiable
  issue in actual `naba` source. Reports that appear to be unverified AI output will
  be closed without response. (This mirrors [Django's](https://docs.djangoproject.com/en/dev/internals/security/)
  and [curl's](https://daniel.haxx.se/blog/2025/07/14/death-by-a-thousand-slops/)
  hard-won stance on AI security-report spam.)

> **In one line:** we don't care that a machine helped — we care that a human
> understands it, stands behind it, and that it passes every check.
